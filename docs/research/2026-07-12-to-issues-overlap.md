# `to-issues` overlap and provenance audit

Date: 2026-07-12

Question: How does Recruitr's `to-issues` differ from the existing public `to-tickets` workflow, and what—if anything—should the public catalog preserve?

## Decision

Do **not** publish Recruitr's `to-issues` as an independent skill. It is a lightly adapted copy of Matt Pocock's former `to-issues`, and upstream has since deleted that skill in favor of the more capable, tracker-agnostic `to-tickets`. Shipping Recruitr's copy would duplicate an actively maintained public workflow while dropping upstream improvements. [Recruitr vendoring commit](https://github.com/cdowell09/recruitr/commit/75ed17774b423b02275f7d9a7a66cf3e9b523e4a) [upstream unification commit](https://github.com/mattpocock/skills/commit/386d4ff719a7c420ad1454232d0436b01f1b8c17) [current upstream `to-tickets`](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/skills/engineering/to-tickets/SKILL.md)

Preserve one policy as an optional extension to ticket decomposition: classify each proposed slice as **agent-doable** or **human-gated**, then map that disposition through capability configuration to `ready-for-agent` or `ready-for-human`. This is useful to the planned agent and human board workers, but it is a routing policy—not enough original behavior to justify a fork of the whole decomposition skill. Recruitr's terminology and label mapping were introduced in its later terminology commit; the underlying HITL/AFK distinction was already present upstream. [Recruitr terminology commit](https://github.com/cdowell09/recruitr/commit/e05a0bdb69665e27c1fff338dc0ca178f14d36c7) [upstream snapshot containing HITL/AFK](https://github.com/mattpocock/skills/blob/ff3ee1dda50d636b4e25b1ba269d25d6d5784164/skills/engineering/to-issues/SKILL.md)

Treat GitHub-native dependencies as a tracker-adapter rule, not as a Recruitr-owned skill feature. Current upstream already prefers native relationships on real trackers and falls back to textual edges only when necessary; GitHub now documents both dependency and sub-issue APIs, and current `gh` exposes dependency flags directly. [current upstream `to-tickets`](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/skills/engineering/to-tickets/SKILL.md) [upstream GitHub tracker adapter](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/skills/engineering/setup-matt-pocock-skills/issue-tracker-github.md) [GitHub dependency docs](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/creating-issue-dependencies) [GitHub sub-issue API](https://docs.github.com/en/rest/issues/sub-issues)

## Provenance

### Source facts

1. The Recruitr skill identifies itself as a copy shadowing `mattpocock-skills/to-issues` and says its one intended difference is GitHub-native `addBlockedBy` relationships instead of a `## Blocked by` body section. Its creation commit likewise says the skill was vendored and edited for native dependencies. [Recruitr source](https://github.com/cdowell09/recruitr/blob/ebb69161e20e22349d210eb7d56a5c481716c5e6/.claude/skills/to-issues/SKILL.md) [Recruitr vendoring commit](https://github.com/cdowell09/recruitr/commit/75ed17774b423b02275f7d9a7a66cf3e9b523e4a)

2. The upstream revision current before Recruitr's vendoring contains the same workflow structure: gather context, optionally explore the codebase, form tracer-bullet vertical slices, classify HITL/AFK work, quiz the user, publish blockers first, use the same issue template, and leave the parent issue unchanged. Recruitr retained that process and replaced the body-based dependency representation with its own sixth step. [upstream snapshot](https://github.com/mattpocock/skills/blob/ff3ee1dda50d636b4e25b1ba269d25d6d5784164/skills/engineering/to-issues/SKILL.md) [Recruitr source](https://github.com/cdowell09/recruitr/blob/ebb69161e20e22349d210eb7d56a5c481716c5e6/.claude/skills/to-issues/SKILL.md)

3. Recruitr later renamed upstream's `HITL`/`AFK` classification to `human-gated`/`agent-doable` and made the mapping to `ready-for-human`/`ready-for-agent` explicit. The diff changes terminology and labels, not the underlying classification behavior. [Recruitr terminology commit](https://github.com/cdowell09/recruitr/commit/e05a0bdb69665e27c1fff338dc0ca178f14d36c7) [upstream HITL/AFK source](https://github.com/mattpocock/skills/blob/ff3ee1dda50d636b4e25b1ba269d25d6d5784164/skills/engineering/to-issues/SKILL.md)

4. Upstream later added native parent/sub-issue and blocking relationships to `to-issues`, with body sections only as fallbacks. That removed Recruitr's principal behavioral differentiation, while doing so through tracker configuration rather than a hard-coded repository. [upstream native-relationship commit](https://github.com/mattpocock/skills/commit/0172e61e57c9cae07955e8e30a0f010cee3dbe54) [last upstream `to-issues`](https://github.com/mattpocock/skills/blob/c5a4a8c2e966e28628f56d1bfef07f401d399df0/skills/engineering/to-issues/SKILL.md)

5. Upstream then merged `to-plan` and `to-issues` into `to-tickets` and deleted `to-issues`. The replacement makes blocking edges the common abstraction for both sequential local files and parallel work on a real tracker. [upstream unification commit](https://github.com/mattpocock/skills/commit/386d4ff719a7c420ad1454232d0436b01f1b8c17) [current upstream documentation](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/docs/engineering/to-tickets.md)

6. Matt Pocock's repository is MIT-licensed. Any publication that copies a substantial portion of the upstream skill must retain its copyright and permission notice. [upstream license](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/LICENSE)

### Inference

The Recruitr file is a derivative adaptation, not an independently originated decomposition workflow. The only Recruitr-authored behavior with lasting product value is the explicit routing disposition and its configured label mapping. Even that is an extension of an upstream classification rather than a clean-room invention. The safest public design is therefore composition or a small documented extension around upstream `to-tickets`, with attribution, rather than a renamed copy.

## GitHub-native dependency changes

### Source facts

- Recruitr's May vendoring commit replaced Markdown parsing and repeated blocker lookups with structured GraphQL `blockedBy`/`blocking` connections and `addBlockedBy`; it deliberately retired dual-writing and made the native graph the sole source of truth for its board planner. [Recruitr vendoring commit](https://github.com/cdowell09/recruitr/commit/75ed17774b423b02275f7d9a7a66cf3e9b523e4a)
- Upstream's later `to-issues` revision preferred native sub-issue and blocking relations on any capable tracker and retained body sections only as fallbacks. [upstream native-relationship commit](https://github.com/mattpocock/skills/commit/0172e61e57c9cae07955e8e30a0f010cee3dbe54) [last upstream `to-issues`](https://github.com/mattpocock/skills/blob/c5a4a8c2e966e28628f56d1bfef07f401d399df0/skills/engineering/to-issues/SKILL.md)
- Current upstream `to-tickets` carries that policy into a tracker-neutral abstraction. Its GitHub setup template currently documents the REST `dependencies/blocked_by` endpoint, while its local tracker writes textual `Blocked by` edges to individual ticket files. [current `to-tickets`](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/skills/engineering/to-tickets/SKILL.md) [GitHub adapter](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/skills/engineering/setup-matt-pocock-skills/issue-tracker-github.md) [local adapter](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/skills/engineering/setup-matt-pocock-skills/issue-tracker-local.md)
- GitHub's current official interfaces support dependency creation and inspection directly through `gh issue create`, `gh issue edit`, and `gh issue view`, as well as REST and GraphQL APIs. [GitHub CLI dependency docs](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/creating-issue-dependencies) [REST dependency API](https://docs.github.com/en/rest/issues/issue-dependencies) [GraphQL issue mutations](https://docs.github.com/en/graphql/reference/issues#addblockedby)

### Inference

A public GitHub adapter should prefer the documented `gh issue` dependency flags because they avoid embedding GraphQL node-ID lookup code in the skill. REST or GraphQL can remain capability fallbacks. The semantic contract should be “record a native blocked-by edge”; the transport belongs in repository-owned capability configuration. This preserves Recruitr's structured, UI-visible, single-source-of-truth intent without retaining its owner, repository, script, or API-shape coupling.

## What to preserve, adapt, and drop

### Preserve as shared policy

- **Disposition:** `agent-doable` versus `human-gated`, mapped to configured triage roles. Keep it optional because upstream `to-tickets` intentionally emits agent-ready tickets. [Recruitr terminology commit](https://github.com/cdowell09/recruitr/commit/e05a0bdb69665e27c1fff338dc0ca178f14d36c7) [current upstream publication rule](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/skills/engineering/to-tickets/SKILL.md)
- **Source traceability:** optionally show which user stories a slice covers during approval when the source has user stories. [Recruitr source](https://github.com/cdowell09/recruitr/blob/ebb69161e20e22349d210eb7d56a5c481716c5e6/.claude/skills/to-issues/SKILL.md)
- **Native-edge semantics:** on GitHub, one native dependency graph should be authoritative; do not dual-write body sections. This should be expressed by the GitHub tracker adapter. [Recruitr vendoring commit](https://github.com/cdowell09/recruitr/commit/75ed17774b423b02275f7d9a7a66cf3e9b523e4a) [GitHub dependency docs](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/creating-issue-dependencies)

### Adopt from current upstream

- Prefactoring before slicing, one-context-window sizing, the wide-refactor expand–contract exception, generic frontier semantics, native sub-issues when available, and local/real tracker adapters are all present upstream and absent or less complete in Recruitr. [current upstream `to-tickets`](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/skills/engineering/to-tickets/SKILL.md)

### Drop

- The hard-coded `cdowell09/recruitr` GraphQL example, `board-plan.mjs` coupling, and local triage-doc path are repository-specific implementation details. [Recruitr source](https://github.com/cdowell09/recruitr/blob/ebb69161e20e22349d210eb7d56a5c481716c5e6/.claude/skills/to-issues/SKILL.md)
- The `to-issues` name and independent package should not survive; upstream now uses the broader `to-tickets` contract and has removed `to-issues`. [upstream unification commit](https://github.com/mattpocock/skills/commit/386d4ff719a7c420ad1454232d0436b01f1b8c17)
- Fixed counts in the issue-template example and other wording-only differences have no differentiated workflow value. [Recruitr source](https://github.com/cdowell09/recruitr/blob/ebb69161e20e22349d210eb7d56a5c481716c5e6/.claude/skills/to-issues/SKILL.md) [current upstream `to-tickets`](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/skills/engineering/to-tickets/SKILL.md)

## Consequence for the publication plan

Remove `to-issues` from the list of independent publication candidates. In its place, the catalog plan should choose one of these implementation shapes later:

1. **Preferred:** depend on or recommend the independently installable upstream `to-tickets`, and define the catalog's human-gate disposition as a small composable policy consumed by ticketing and board-worker skills. The upstream repository documents installation via `npx skills add mattpocock/skills --skill=to-tickets`. [upstream `to-tickets` documentation](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/docs/engineering/to-tickets.md)
2. If composition cannot express the required classification, publish a thin, clearly attributed extension that changes only disposition and configured label mapping. Do not copy the full skill without retaining the upstream MIT notice. [upstream license](https://github.com/mattpocock/skills/blob/19c50d50fd1f1fd73cffc4de61a1aabee4cf8728/LICENSE)

This audit does not choose the extension mechanism; it establishes that a standalone Recruitr-derived `to-issues` is redundant and that any preserved behavior must be isolated from the upstream decomposition workflow.
